## 3. Project Cards

### Section subtitle
- Problem → Judgment → Decision → System → Outcome

### ByteDance · Meego NL2SQL
- **Product Judgment:** Root cause: NL2SQL is unreliable in production because “correct” has no reproducible definition; optimization is guesswork. Judgment: Make evaluation observable first, then optimize models and prompts; otherwise iteration has no basis. So we made evaluation a product capability and used attribution to show “where it fails.”
- **Decision:** End-to-end machine eval + error attribution first → accuracy 58% → 95%+; architecture evolved from single-turn to multi-role.

### Euler · Financial Report AI
- **Product Judgment:** Root cause: Analysts find comparable companies in large report sets via manual search—inefficient and inconsistent. Judgment: If we pick a model then justify it, we bias the answer; we need a level comparison first, then trade off accuracy, latency, and traceability. So we chose FinGPT + Fin-R1 for retrieval quality and explainability.
- **Decision:** Baseline library + four-approach comparison → 0→1 validation; core scenarios >80% accuracy.

## 4. MVP → Scalable System

- **Title:** From MVP to scalable system
- **Intro:** Why each stage exists: so the next decision has evidence, not more features.

| Stage | What | Why this stage exists |
|-------|------|------------------------|
| MVP | Minimal path to core value (usable) | Validate “right” before “big”; avoid investing in the wrong direction. |
| V1 | Automated evaluation (measurable) | Without observable metrics, iteration is blind; evaluation is the basis for control. |
| V2 | Attribution dimensions (diagnosable) | Knowing “which type of error” enables targeted fix; otherwise it’s trial and error. |
| V3 | Multi-role collaboration (scalable) | Single-point capability has a ceiling; composable subsystems handle complexity. |
| V4 | Methodology and framework (repeatable) | Codify experience so the next project starts at V2, not zero. |
